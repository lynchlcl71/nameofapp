Learning Goals

  • Describe which actions will be in the comments controller
  • Create your comments controller and views
  • Install and enable the Raty plugin for star ratings

Create the Comments Controller

Let’s go ahead and create our comments controller. We won’t be using the scaffold generator here because we don’t need any views specifically for comments. The comment form and listing will be added to the product details page, and the destroy action doesn’t need a view.

$ rails generate controller comments

So if we open up the controller, we just need to add the methods for create and destroy:

class CommentsController < ApplicationController

  def create
  end

  def destroy
  end

end

Nested Resources

Now, we will introduce a new concept: because these user reviews are associated with a specific product, we will treat the “comments” resources as a subset of “products”. This is called a nested resource.  We define this in our routes file simply like so:

resources :products do
  resources :comments
end

Note: earlier in this course, we showed that you can list multiple CRUD resources on the same line, so you may already have this in your routes file:

resources :products, :users

If that’s the case, just declare the users resources and products resources on separate lines in order to create the nested resource block above:

resources :products do
  resources :comments
end
resources :users

What nested resources allow us to do is to connect comments to products (similar to how we connected orders and users earlier), so you can reference them like so:

@product.comments

Or when we create a comment, we can create it as a child record of a particular product. That means we can build our comments#create method like so, using @product.comments.new:

def create
  @product = Product.find(params[:product_id])
  @comment = @product.comments.new(comment_params)
  @comment.user = current_user
  @comment.save
  redirect_to product_path(@product)
end

On that second line of the method we are using the comment_params method to set the attributes of the comment object. The comment_params variable is just the portion of the params which are contained within the "comment" block. So, the entire parameters passed to the action look like this (in your log):

Parameters: {"utf8"=>"✓", "authenticity_token"=>"FzUH...", "comment"=>{"body"=>"i really love this product!", "rating"=>"5"}, "commit"=>"Submit", "product_id"=>"3"}

And the comment_params method is a shortcut that can be used to return this:

"comment"=>{"body"=>"i really love this product!", "rating"=>"5"}

Alternatively you can use the standard hash key syntax, which is how this was done in Rails until Rails 4 introduced strong parameters :

@comment = @product.comments.new(params[:comment])
Strong Parameters

Let's take a look at the comment_params method, which should be in the private block at the bottom of your comments controller:

private

  def comment_params
    params.require(:comment).permit(:user_id, :body, :rating)
  end

The "model_params" methods are typically auto-generated when you run the Rails generator for scaffolding. It lists the attributes of a model that a user is allowed to update; if you omit certain attributes, it can also be used to prevent those attributes from being updated on an object publicly (normally to prevent a hacker or nefarious user from updating data that they shouldn't be able to access)-- but you can still edit those attributes via the console if you want. If at this point you get an error in your app that comment_params is undefined, or if you manually created the controller file, you will need to define the comment_params method. To do that just take the code above and put it at the bottom of your CommentsController file before the last end. Take a look at the Rails documentation on strong parameters for more info.

Views

Let’s add a partial called _new_comment.html.erb to our products show view so that a form displays underneath a product (if a user is signed in):

<% if signed_in? %>
  <h4>Add a review:</h4>
  <%= form_for([@product, @product.comments.build]) do |f| %>
    <p>
      <%= f.label :body, "Comments" %><br>
      <%= f.text_area :body %>
    </p>
    <p>
      <%= f.label :rating %><br>
      <%= f.text_field :rating, class: "rating" %>
    </p>  
    <p>
      <%= f.submit "Submit" %>
    </p>
  <% end %>
<% end %>

That will look something like this:

And then we can add another partial which will display the listing of reviews. First we will retrieve the comments to display on the show page, in our products controller:

# GET /products/1
# GET /products/1.json
def show
  @comments = @product.comments.order("created_at DESC")
end

And then create the partial -- let's call it _comments.html.erb and put it also in the views/products folder --, which we will also display on the products show page:

<div class="product-reviews">
    <% @comments.each do |comment| %>
        <div class="row">
            <hr>
            <p><%= comment.user.first_name %> <small><em><%= "#{time_ago_in_words(comment.created_at)} ago" %></em></small></p>
            <p><%= "Rating: #{comment.rating}/5" %></p>
            <p><%= comment.body %></p>
        </div>
    <% end %>
</div>

Which will end up looking like this, when users submit some reviews:

Notice the way we display the timestamp. We’re using a Rails helper called time_ago_in_words. This is a helpful way to display a database timestamp in a human-readable format, which approximates when a record was created. Also, we are going to format the rating so it will display a star rating system (5 out of 5) as opposed to entering and displaying an integer.

Rating System

Now we’ll build out the rating functionality. The finished product will look something like this:

There are several pre-packaged solutions available on the web for star rating systems. One excellent one is the ratyrate gem, which leverages a jQuery plugin called Raty and adds an implementation for a Rails app. This gem only takes a few steps to install and provides rating abilities for any model and for any number of attributes you decide to declare.

But since we’ve already laid the foundation of our rating system, let’s skip the gem and keep going with building it on our own (you’re here to learn, right?). Let’s use the jQuery plugin Raty for the front end component. It’s relatively easy to install: go to the plugin site, click the "releases" tab and under "Downloads" and then on"Source code (zip)" and extract the necessary files into your assets folders:

  • jquery.raty.js → assets/javascripts/

  • jquery.raty.css → assets/stylesheets

  • images/ → assets/images

Then, according to the plugin instructions you only need to declare a specific element to have the rating styles applied to it. We’re going to declare a div with class “rating” where we want our rating selection to appear:

...
  <p>
    <%= f.label :body, "Comments" %><br>
    <%= f.text_area :body %>
  </p>

  <p>
    <div class="rating"></div>
  </p>

  <p>
    <%= f.submit "Submit" %>
  </p>
...

and then initialize the plugin on this element. You can put this function inside a new JavaScript file in your assets/javascripts folder (any new JS files you put into this folder should be included by your application automatically). We will call this file site.js (with the intention being that this file will contain any site-wide functions):

$(document).on('turbolinks:load', function(){
    $('.rating').raty();
});

What is the "Turbolinks" gem?
Because of the turbolinks-gem (a gem that's already included in Rails by default) the line $(document).ready(function(){ ... }); doesn't work as on your previous projects. Turbolinks speed up the page load time when traveling around the different pages of your Rails app. Turbolinks 5 is the latest version and was released with Rails 5. It is a significantly improved version - compared to Turbolinks 4. You can really notice an obvious difference in page load time with Turbolinks. Because of that, the pages don't always do a full document refresh and $(document).ready(function(){ ... }); doesn't get loaded again.
$(document).on('turbolinks:load', function () { // All your code here }); is a simple workaround to recognize each new page load in Rails 5 and up. If your app is still on an older version of Rails you have use $(document).on('ready page:load', function () { // All your code here }); instead! (You can check your Rails version in the Gemfile or your app.)
There is one little thing to keep in mind with this: If you have any other event listeners running on $(document) itself (e.g. $(document).on("click", ...) that one should not be inside the $(document).on('turbolinks ... function. Instead, it should be next to it (either below or above).

If you try to look at your form now, the plugin should be loading, but the images will be broken. This is because we need to specify where the image assets are found in your Rails app. Modify the second line of your site.js script to reflect that:

$('.rating').raty( { path: '/assets' });

You should now see your rating system appear:

But if you try to submit a review now, it won’t save the rating properly. This is because the plugin is passing the rating integer as a hidden field called “score”, whereas our model attribute is called “rating”. Because “rating” is one of the comment parameters (all of the fields in the form are included in the comment parameters) and the name of the attribute, the field should really be named “comment[rating]”. Just like the body field is named “comment[body]”. We can modify the name of the hidden field with this addition when we call the plugin:

$('.rating').raty( { path: '/assets', scoreName: 'comment[rating]' });

Now, when you save a review with a rating it should appear in the listing below, as a number.

But we want the star rating to display there, instead of the integer value. We need slightly different functionality: if you look in the plugin documentation, you’ll see that you can set the stars to “read only” which means the plugin will display a row of stars that can’t be set. You can also add a “data-score” attribute to dynamically populate the number of stars. Since these will be different functionally than the other stars in the form, we should declare them as a separate class “rated” (in _comments.html.erb):

<div class="rated" data-score="<%= comment.rating %>"></div>

and initialize the plugin on that element with these different functions, in site.js:

$(document).on('turbolinks:load', function(){
    $('.rating').raty( { path: '/assets', scoreName: 'comment[rating]' });
    $('.rated').raty({ path: '/assets',
      readOnly: true,
      score: function() {
        return $(this).attr('data-score');
      }
    });
});

Now our ratings system only needs one more step, which is calculating the product’s average rating that we will display on the product detail page. Because each product will have a unique rating, it will be an attribute of the product, so we will declare it as a method on the Product model. We won’t need to store it in the database in this case because we can calculate it fairly simply, and it will change quite often; also, we only need it for display purposes. Active Record has some built-in calculation methods which we can take advantage of for this functionality — specifically the average method. This method takes all the rows of a specific database field and calculates an average of their values. It accepts the database field name as a parameter. Try it out in your Rails console to see how the method is going to work:

> p = Product.first
=> #<Product id: 1, …
> p.comments.average(:rating)
=> #<BigDecimal:7f7fdc5fce00,'0.3666666666 66667E1',27(36)>

Because the average value will rarely be a whole number, the average method always returns a floating point number — specifically, a floating point number represented as a Ruby "BigDecimal" object. You'll notice this object contains several data attributes, but we are only concerned with the numerical value. To extract the value from the BigDecimal object, call the method to_f (“to float”):

> p.comments.average(:rating).to_f
=> 3.66666666666667

So now the next step is to incorporate this into a new method that we can call on any Product instance. We can add an instance method to the product.rb model file, which will enable us to call @product.average_rating and get a decimal number back:

def average_rating
  comments.average(:rating).to_f
end

It's as simple as that! Now to display it in our view, we add this to show.html.erb:

<p>Average Rating: <div class="rated" data-score="<%= @product.average_rating %>"></div> </p>
